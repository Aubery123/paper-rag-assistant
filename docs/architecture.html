1 · 系统定位

研究生每天啃大量论文，痛点是"跨多篇找信息慢、出处记混、写综述要反复对比"。本项目导入研究领域的论文（PDF / arXiv），构建可问答知识库，回答时给出精确到篇 / 节 / 页的引用溯源，并支持多篇对比。

它刻意与作者的另一项目（多 Agent 痛点挖掘系统）互补不重复：那个主打多 Agent 编排，本项目主打 RAG 深度 + 评测驱动——不止"能跑"，而是建评测集量化每一处改进。

4 · 关键模块拆解

📄loader · PDF 解析ingest/loader.py

逐页解析，输出带 {paper_id,title,section,page} 的结构化文本。章节识别两级兜底：优先用 PDF 内置目录 get_toc()（按一级条目数自适应"标题 / 章节"层级），无目录时正则识别章节标题行。标题解析优先级：TOC 根 > 元数据 > 首页最大字号 > 文件名。

✂️chunker · 结构化切分ingest/chunker.py

按页切分，使每个 chunk 只归属一页 → 引用页码绝对精确。递归分隔符（段落→行→句→字符）尽量在自然边界断开，相邻 chunk 带字符级重叠，过短尾块并入前块。每个 chunk 继承所在页元数据 + 唯一 chunk_id。

🔎hybrid · 混合检索retrieve/hybrid.py

向量路（百炼 embedding + Qdrant 余弦）与关键词路（rank-bm25，中文逐字 / 英文整词分词）并行召回，用 RRF 按各路排名倒数加权融合。BM25 索引进程内懒加载缓存。

🎯reranker · 重排retrieve/reranker.py

对混合召回的候选用百炼 gte-rerank-v2 精排（httpx 直连原生端点），返回按相关度排序的 top-k。

💬answerer · 引用作答generate/answerer.py

检索链路定型为 召回(dense|hybrid, k=20) → 重排 → top_k，拼装带编号资料喂给 qwen，要求每个论断标注 [编号] 并可回溯到篇 / 节 / 页。提供流式与非流式两种。检索方式与是否重排为开关，供消融。

📊eval · 评测eval/

评测集含问题 + 参考答案 + 页级金标准。指标分两层：检索层（确定性、无 LLM）页级 Recall@k / MRR；生成层（RAGAS 风格 LLM 判分）忠实度 / 答案相关性 / 上下文召回。run_eval.py 对 4 配置消融出双表。

🌐api / app · 服务与界面api.py · app.py

FastAPI 暴露 /health、/ask、/ask/stream（SSE 逐 token + sources 事件）；Streamlit 聊天界面带流式渲染、引用展示、多篇对比与检索开关。

5 · 核心机制详解

5.1 RRF 融合

对每一路检索结果按排名计算 score = Σ 1/(rrf_k + rank)，两路相加后排序。无需归一化不同路的分数尺度，鲁棒且实现简单。同时出现在两路的 chunk 自然获得更高融合分。

5.2 引用溯源

引用的根基是每个 chunk 必带 {paper_id,title,section,page}。检索结果编号后注入提示词，模型作答时标 [编号]，前端 / CLI 再把编号映射回"《论文》· 章节 · 第 N 页"。按页切分保证页码无歧义。

5.3 评测分层

把检索质量与生成质量解耦衡量：检索层用确定性的页级 Recall@k / MRR（零 LLM、零噪声、近乎零成本）；生成层用 LLM-as-judge 的 RAGAS 风格指标。消融对比 纯向量 / +重排 / 混合 / 混合+重排 四配置。

6 · 开发中遇到的问题与解决

问题 1 · 章节识别

PDF 内置目录（TOC）层级语义不一致：部分论文一级条目是论文标题，部分一级条目直接是章节（Introduction…）。

解决：按一级条目数量自适应——恰 1 个则视其为标题、章节取 level≥2；多个则一级即章节、取 level≥1。无 TOC 的论文（如 Flash）退回正则识别标题行兜底。

问题 2 · Windows 编码

控制台打印中文 / 特殊字符报 UnicodeEncodeError（GBK）。

解决：各 CLI 入口 sys.stdout.reconfigure(encoding="utf-8", errors="replace")。

问题 3 · 纯向量召回单一

宽泛的对比类问题被某一篇的"相关工作"段垄断 top-k，召回多样性不足，答案偏。

解决：引入 BM25 + 向量 + RRF 混合检索，召回从单篇扩到跨多篇；再加重排提升精度。这是检索增强价值最直观的一次体现。

问题 4 · API 形态差异

百炼重排不在 OpenAI 兼容端点上，openai SDK 调不到。

解决：用 httpx 直连 DashScope 原生 rerank 端点；嵌入 / 作答仍走兼容端点。

问题 5 · 评测指标无区分度（关键经验）

① LLM 判分单次调用方差大、题量少时信号被噪声盖过；② 篇级 Recall / MRR 在 5 篇小语料上饱和（MRR=1、Recall@20=1），分不出配置优劣。

解决：改用页级金标准（按章节结构标注答案所在页）+ 确定性检索指标，并把检索层与生成层指标分开。页级后恢复了区分度。

7 · 设计取舍与限制

取舍

全链路走 API、不下载本地模型。嵌入 / 作答 / 重排都用百炼，省去 2GB+ 本地模型与 CPU 慢推理；代价是放弃了本地开源模型与其自带的 sparse 词权重——混合检索的 sparse 路改用纯 Python 的 BM25 补上。

诚实的评测结论

当前评测语料仅 5 篇且主题高度同质（联邦学习 + 概念漂移），纯向量检索已接近天花板，混合 / 重排的增益不明显；页级金标准由章节结构标注、尚待人工核准。这一结论本身即评测驱动的产出——增益需在更大、更异质的语料上、核准金标准后复跑才可信。评测框架是真正的交付物。

8 · 运行方式

docker compose up -d qdrant          # 起向量库
pip install -r requirements.txt
cp .env.example .env                 # 填 DASHSCOPE_API_KEY（百炼）

python main.py ingest ./data/papers/                 # 导入论文
python main.py ask "FLASH 在服务器端如何做漂移感知优化？"   # 提问（带引用）
python main.py eval --limit 2                         # 跑评测（测试用）

uvicorn src.api:app --reload                         # 起 API（/ask、/ask/stream）
streamlit run app.py                                 # 起聊天界面
